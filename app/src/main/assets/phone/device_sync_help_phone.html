AppSyncr Help

It is important to remember that AppSyncr only knows about those devices you have installed the app onto. If AppSyncr not installed on a device, it will not know about that device nor the apps installed upon it.

The phone/tablet UI is a nav drawer set up allowing you to slide a navigation drawer from the left to choose which items you wish to view.

   

Main UI:

Devices - This row lists all known devices attached to your account. You can browse the apps installed on the device or clone a remote device’s app load to your local system.

Local Apps - This row lists the apps on your local device. Note that we filter out system apps (can change in settings panel). You can uninstall apps on this row.

Missing Apps - This row lists the apps which exist on other compatible devices but not on your local device. You can install apps on this row.

Unique Apps - This row lists the apps which are installed only on your local device. You can uninstall apps on this row.

App Superset - This row lists a superset of all the apps installed on all of your compatible devices. Apps can be installed/uninstalled on this row.

Remote Devices - A row per remote device which shows the apps installed on that device.

   

   

Options Menu:

Force Sync - Forces a read of the cloud database along with a read of all apps on the device.

Disable/Enable Syncs - Disables updating of the cloud and the local content provider for app changes. Thus UI will no longer change when syncs are disabled.

Help - Shows this help.

Settings - Shows the settings page.

Settings:

Ignore system apps - when set, apps marked as system apps (which can’t be uninstalled), will be ignored by AppSyncr.

How it all works:

This is not a complete section but hits the highlights.

First, all devices are identified through the serial number on the device. Note this means there is an implicit assumption that a user has a unique serial number for all their devices. While not guaranteed, it is extremely unlikely to have 2 duplicate serial numbers for production devices.

Second, all apps are tracked via their package name. Again, if this is a play store app, this is guaranteed. However, if you are an app developer, you may have two apps with the same package name. AppSyncr will probably get confused.

Third, AppSyncr does support apps not on play store. Before we issue any install intents, we do look to the play store to see if that package exists.

Now, the how it works section. The summary is we have a traditional UI <-> Content Provider model. However, on the back end of the CP, we also attach through a service to a Firebase cloud database.

The service - we start a service on boot complete which attempts to log into firebase. This watches for data changes in the cloud and, when one occurs, replicates the change to our content provider. And when the content provider changes from local data, this same service replicates the CP changes back up to the cloud.

Local data changes come from two sources - either the app itself is running *or* the user installs/uninstalls an application while the app is not running. We have a broadcast receiver installed which will catch app package events. This receiver will hand off the event to an intent service which will then update our own local CP provider. The service (mentioned above) will be called by the CP to then replicate to the cloud.

The broadcast receiver is also used for batch installs/uninstalls. We will set up an arraylist of apks to install/uninstall and kick off an intent to process the first one. Once that intent fulfilled, the package install broadcast will trigger our broadcast receiver which will repeat the install intents in a loop until the list is drained.

Data is stored in 3 basic groups - devices, apps and images (banners/icons for the app). Devices are just that - it is a CP for devices only and referenced by serial number. Apps are stored referenced by serial/apk. Images for apps are grabbed from the local apk itself. For local use, the local apk will supply the image. However, as we grab apk banners, we will also upload these to the cloud firebase storage if they don’t already exist up in the cloud. Then remote devices will pull these images as needed when displaying info about a remote app that does not exist on the local system. A reference to these download urls is maintained in a third local CP.

The data model is such that each device has a full replicated copy of what exists in the cloud. And local changes replicated to the cloud as soon as they happen (so every device has a functional copy of the database).

Finally, in the cloud, we store the three databases by user token. When you first log on, a random (but unique) UID is generated and given to us by google. We use this UID to store the device/apk/image data for a user up in the cloud.
